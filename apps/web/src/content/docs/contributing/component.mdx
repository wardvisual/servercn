# Contributing Components to Servercn

Servercn is a backend component registry for Node.js and TypeScript projects.  
This guide explains **exactly how to create, structure, validate, and submit a reusable component** to the registry.

> This guide is strictly for contributing registry components — not core CLI or documentation changes.

<br/>
---

## Step-by-Step Contribution Guide

<Steps>
<Step>
Fork & Clone the Repository
</Step>
Fork the Servercn repository and clone it locally:
[GitHub Link](https://github.com/AkkalDhami/servercn)

```bash
# Clone on your local machine
git clone https://github.com/<your-username>/servercn.git

# Navigate to project directory
cd servercn

# Create a new Branch
git checkout -b my-new-branch

# Install dependencies
npm install

# Run a workspace
npm run dev:app # app
npm run dev:cli # cli
```

<Step>
Understand Component Structure
</Step>
Each component must follow the registry structure:

```
packages/
  registry/
    component/
      component-name.json
```          

<Step>
Add Registry
</Step>
Each component must follow the registry structure:

#### 1. Add Registry Without Variant

```json
{
  "$schema": "https://servercn.vercel.app/schema/component.registry.json",
  "slug": "test-component",
  "runtimes": {
    "node": {
      "frameworks": {
        "express": {
          "dependencies": {
            "dev": ["express"],
            "runtime": ["@types/express"]
          },
          "templates": {
            "feature": "test-component/feature",
            "mvc": "test-component/mvc"
          },
          "env": ["TEST"]
        }
      }
    }
  }
}
```

#### 2. Add Registry With Variant

```json {10-21}
{
  "$schema": "https://servercn.vercel.app/schema/component.registry.json",
  "slug": "test-component",
  "runtimes": {
    "node": {
      "frameworks": {
        "express": {
          "prompt": "Select the variant: ",
          "variants": {
            "test-variant": {
              "label": "Label that shown in the cli",
              "dependencies": {
                "dev": ["express"],
                "runtime": ["@types/express"]
              },
              "templates": {
                "feature": "test-component/feature",
                "mvc": "test-component/mvc"
              },
              "env": ["TEST"]
            }
          }
        }
      }
    }
  }
}
```

##### I. $schema

- Points to the official Servercn [component](https://servercn.vercel.app/schema/component.registry.json) schema.
- Ensures your configuration is validated automatically.
- If this is missing or incorrect, the component will fail validation.
- Always use the official schema URL.

```json
{
  "$schema": "https://servercn.vercel.app/schema/component.registry.json",
}
```

##### II. slug

- Unique identifier of the component.
- Must be kebab-case.
- Used internally by CLI commands:

<PackageManagerTabs command="npx servercn-cli add test-component" />

```json
{
 "slug": "test-component"
}
```

##### III. runtimes

Defines which runtime environments support this component.

```json
"runtimes": {
  "node": { ... },
  "bun": { ... }
}
```
<LNote>
- Currently, node is the primary runtime.
- Future runtimes (e.g., bun, deno) may be added.
- Your component must not assume compatibility with unsupported runtimes.
</LNote>

##### IV. frameworks

Specifies which frameworks the component supports.

```json
"frameworks": {
  "express": { ... }
}
```
<LNote>
- Only define frameworks your templates actually support.
- If you declare express, you must provide valid Express-compatible templates.
- Do not declare frameworks without implementation support.
</LNote>

##### V. prompt

```json
{
  "prompt": "Select the variant: "
}
```

<LNote>
- Displays a selection message in the CLI.
- Used when your component has multiple variants.
- Keep prompt concise.
- Avoid explanations inside the prompt.
- Prompt should describe a decision (library, strategy, mode).
</LNote>

Example CLI output:

```
? Select the variant:
> test-variant
  test-variant2
```

##### VI. variants

Each key under <Code>variants</Code> represents a selectable implementation option.

```json
"variants": {
  "test-variant": { ... }
}
```

<LNote>

- Key must be kebab-case.
- Each variant must define:
   <Code>label</Code>, <Code>templates</Code>
- Optional but recommended:
   <Code>dependencies</Code>, <Code>env</Code>
</LNote>

##### VII. label

```json
{
  "label": "Label that shown in the cli"
}
```
<LNote>
- Human-readable name shown in CLI.
- Should clearly describe the variant.
</LNote>

Example for <Code>email-provider</Code>:
```json
{
  "label": "Select an email provider: "
}
```

##### VIII. dependencies

Defines packages automatically installed when the component is added.

```json
{
  "dependencies": {
    "dev": ["express"],
    "runtime": ["@types/express"]
  }
}
```

```
| Field     | Meaning                  |
| --------- | ------------------------ |
| `runtime` | Production dependencies  |
| `dev`     | Development dependencies |
```

<LNote>
- Only include packages required by your templates.
- Do not include unnecessary peer dependencies.
</LNote>

##### IX. templates

Maps architecture types to template folders.

```json
{
  "templates": {
  "feature": "test-component/feature",
  "mvc": "test-component/mvc"
  }
}
```

<LNote>
- If you declare both, you must implement both.
- Template folder paths must exist inside: 
- <Code>packages/templates/node/express/component/test-component/</Code>
</LNote>

Example:

```bash
packages/
  templates/
    node/
      express/
        component/
          component-name/
            feature/
            mvc/
```      

##### X. env

Defines required environment variables.

```json
{
  "env": ["NODE_ENV"]
}
```

<LNote>
- Only include truly required variables.
- Use clear naming conventions.
- Avoid default secrets inside templates.
</LNote>

<Step>Add Templates</Step>

Template must follow the folder structure below:

```bash
packages/
  templates/
    node/
      express/
        component/
          component-name
            feature/
            mvc/
```      

<LNote>
- Follow clean separation of concerns
- Match selected architecture style
- Avoid unnecessary dependencies
- Be production-ready
</LNote>

<Step>Test the Component via CLI</Step>

**Before submitting:**

```bash
npm run build:cli
```

<PackageManagerTabs command="npx servercn-cli add test-component --local" />

- <Code>--local</Code>: Used for testing in the local environment.

**Verify:**

<LNote>
- No runtime errors
- Files generate correctly
- Architecture mapping works
- Stack selection works
</LNote>


<Step>Build the Registry Items</Step>

Before opening your pull request, ensure the registry output is generated and up to date.
<PackageManagerTabs command="npx servercn-cli build" />

#### What This Does:

<LNote>
  - Compiles all registry items
  - Validates schema structure
  - Processes component metadata
  - Outputs generated files into: <Code>apps/web/public/sr/</Code>
</LNote>

#### Verification:

<LNote>
  - After running the build command:
  - Navigate to <Code>apps/web/public/sr/</Code>
  - Confirm your component/item appears
  - Ensure generated JSON reflects your changes
</LNote>

> Only proceed with your pull request after confirming the registry output is correct.

<Warning text="Don't forget to run `npx servercn-cli build` before submitting your pull request to ensure the registry output is up to date." />

<Step>Register in registry.json</Step>

Add registry inside:
<Code>apps/web/src/data/registry.json</Code>

```json
{
  "name": "Servercn",
  "homepage": "https://servercn.vercel.app",
  "author": "Akkal Dhami",
  "items": [
    ...
      {
        "slug": "component-slug",
        "title": "Component Title",
        "description": "Component short description",
        "type": "component",
        "status": "stable",
        "docs": "/docs/components/component-slug"
      },
      ....
  ]
}
```

> This configuration is used to generate the docs sidebar and display components in the <Code>/components</Code> route.

<Step>Write Minimal Documentation</Step>

Add mdx documentation inside:

<Code>apps/web/src/content/docs/express/components/component-slug.mdx</Code>

Keep documentation concise and technical.

<Warning text="Don't forget to include `mvc_structure` and `feature_structure` in the front matter."/> 

<Step>Commit Properly</Step>

Use conventional commit format:

> feat(component): add password-hashing component

**Example:**

> feat(security-header): add middleware for configuring security headers with Helmet and CORS


<Step>Submit a Pull Request</Step>

**Your PR must include:**
<LNote>
- Clear title
- Component summary
- Supported stacks
- Testing confirmation
</LNote>

**PR checklist:**
<LNote>
- Schema valid
- Templates structured correctly
- CLI tested
- No unrelated changes
- Clean commit history
</LNote>
</Steps>
